KORStockScan 작업 시 다음 원칙을 기본으로 따른다.

- 이 프로젝트의 최종 목표는 손실 억제가 아니라 기대값/순이익 극대화다.
- 현재 단계가 음수 leakage 제거이더라도, 해석과 제안은 공격적 기대값 개선 관점에서 정리한다.
- 스캘핑 분석에서는 실현손익만 보지 말고 미진입 기회비용까지 함께 해석한다.
- BUY 후 미진입은 latency guard miss, liquidity gate miss, AI threshold miss, overbought gate miss로 분리 분석한다.
- full fill과 partial fill은 동일 표본으로 합치지 말고 별도 해석한다.
- 실전 로직 변경은 한 번에 한 축 canary만 적용하고, 가능하면 shadow-only 및 롤백 가드를 둔다.
- 원인 귀속이 불분명하면 먼저 리포트 정합성, 이벤트 복원, 집계 품질부터 점검한다.
- NULL, 미완료 상태, fallback 정규화 값은 손익 기준으로 사용하지 않는다. 손익 계산은 COMPLETED + valid profit_rate만 사용한다.
- 비교 리포트에서는 왜곡 위험이 있는 손익 파생값보다 거래수, 퍼널, blocker 분포, 체결 품질을 우선 본다.
- 답변은 가능하면 판정, 근거, 다음 액션 순서로 간결하게 정리한다.
- 관련 문서나 체크리스트가 있으면 가능하면 함께 업데이트한다.

작업자동화 원칙:
- 문서가 생성/수정되면 `문서 -> GitHub Project -> Calendar` 자동화 체인을 항상 유지한다.
- 지금 바로 수행하지 않는 작업, 특정 시간에 수행할 작업, 미래의 어느 시점에 다시 확인/실행해야 하는 작업은 반드시 문서에 자동 파싱 가능한 작업항목으로 남기고, 같은 턴에서 `GitHub Project` 등록 대상으로 반영한다.
- 시간 정보가 있는 작업은 가능하면 `Due`, `Slot(PREOPEN/INTRADAY/POSTCLOSE)`, `TimeWindow`까지 함께 확정하고 캘린더 동기화 대상이 되게 유지한다.
- 사용자가 따로 챙겨야 하는 미래 작업을 답변 텍스트에만 남기고 끝내지 않는다. 미래 작업은 원칙적으로 `Project + Calendar`에 올라가 있어야 한다.
- 미래 작업을 새로 만들거나 수정했으면 가능하면 같은 턴에 `sync_docs_backlog_to_project`까지 실행한다. `GitHub Project`/`Calendar` 권한이나 env가 없으면, 무엇이 막혔는지와 사용자가 실행할 1개 명령만 남긴다.
- `Calendar` 동기화까지 가능한 환경이면 `sync_github_project_calendar`도 같은 턴에 이어서 실행해 `Project -> Calendar` 정렬을 완료한다.
- 시간대별 운영 점검, 장전/장중/장후 확인, 익일 확인, 장후 재판정 같은 반복 작업은 가능하면 수동 메모가 아니라 자동 실행(cron/workflow) 또는 최소한 재실행 가능한 wrapper/스크립트로 만든다.
- 자동화 스크립트/cron/workflow를 추가했다면 관련 체크리스트, 운영문서, backlog 문서에도 같은 변경 세트로 반영한다.
- 작업 항목은 자동 파싱 가능 형식으로 작성한다. (체크리스트는 `- [ ]`, 완료는 `- [x]`, 단계는 번호 목록)
- 자동관리 대상 항목은 제목/문구를 불필요하게 자주 변경하지 않는다. (동일 항목 추적 가능성 유지)
- Project 필드는 `Status`, `Due`, `Track`, `Slot(PREOPEN/INTRADAY/POSTCLOSE)`를 표준으로 사용한다.
- `Slot`이 비어 있으면 자동 채움 규칙을 우선 적용하고, 실패 시 원인과 보정값을 문서에 기록한다.
- `Done` 처리는 문서 기준으로 동기화되도록 유지하고, 수동 상태변경 시 문서와 불일치가 생기지 않게 맞춘다.
- 자동화 실행 후에는 최소한 다음을 확인한다: 생성/동기화 건수, `slot_filled`, 실패 항목, 재실행 필요 여부.
- 자동화 실패 시 우회 수동처리보다 먼저 원인 수정(변수/필드명/옵션명/권한)을 우선한다.
- 코덱스 작업지시는 슬롯별 작업지시서(장전/장중/장후) 기반으로 수행한다.
- 자동화 규칙 변경 시 관련 운영문서와 체크리스트를 같은 변경 세트로 함께 업데이트한다.

환경/실행 원칙:
- Python 작업은 기본적으로 프로젝트 `.venv` 가상환경을 사용한다.
- 라이브러리 설치, 업그레이드, 제거처럼 환경을 변경하는 작업은 먼저 사용자 의사결정을 받는다.
- 패키지 설치가 필요하면 왜 필요한지, 설치 대상이 무엇인지, 대안이 있는지를 짧게 설명한 뒤 진행한다.
- 임시 스크립트보다 재현 가능한 명령과 프로젝트 표준 실행 경로를 우선한다.
