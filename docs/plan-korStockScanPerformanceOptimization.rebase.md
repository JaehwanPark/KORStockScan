## 3. Tuning Principles

### 3.1 Performance Decisions And Data Criteria

1. The final goal is not loss suppression, but expected value and net-profit maximization.
2. PnL decisions use only `COMPLETED + valid profit_rate`. Exclude `NULL`, incomplete, and fallback-normalized values from PnL criteria.
3. Do not merge `full fill` and `partial fill`. Fill quality, downstream exit, and PnL are separated.
4. Comparison priority is `trade_count -> funnel -> blocker -> fill_quality -> missed_upside -> PnL`. Do not sum `counterfactual` values with directly realized PnL.
5. Win rate is `diagnostic_win_rate`, not standalone live/canary approval criteria. `primary_ev` owns expected value and net-profit decisions.
6. Simple PnL sum is not EV. Name it `simple_sum_profit_pct`; EV fields must be one of `equal_weight_avg_profit_pct`, `notional_weighted_ev_pct`, or `source_quality_adjusted_ev_pct`.
7. Daily-only values may be used for incidents, safety vetoes, freshness/source-quality, and intraday operational triggers. Edge apply approval must also check rolling/cumulative or `post_apply_version_window`.
8. If attribution is unclear, inspect report consistency, event reconstruction, and aggregation quality before changing strategy thresholds.
9. User-facing tuning performance reports must not lead with a thin headline count as if the count alone were success or failure. For LDM, bucket, active-sim-priority, scalp sim, and swing sim reports, present the result in this order: `decision -> evidence -> next action`. The decision must answer whether the flow/bucket/arm was identified with valid source quality, whether it was actually handed off to the sim policy catalog/PREOPEN env/runtime observation, and what exact blockers remain before real runtime reflection. Evidence must separate complete lifecycle flows, stage-only buckets, sim-auto approvals, active priority matches, EV fields, sample floors, source-quality warnings, submit drought, bridge/live-auto contract status, AI review status, and post-apply attribution. A small `complete lifecycle flow` count is not automatically a failure when catalog/PREOPEN handoff and natural-match semantics are intact; it is early evidence that must be reported with sim application status and real-runtime remaining work. Source-quality unknowns are not described as a current real-trading blocker when the candidate has not reached real-trading authority; they are evidence-quality warnings that must be resolved, excluded, or explicitly marked not-applicable before the same data can support live-auto, real runtime reflection, or full-live approval. Bridge/live-auto reports must not stop at "contract not closed"; they must enumerate each blocker by class (`source_quality`, `sample_floor`, `submit_drought`, `env_mapping`, `runtime_hook`, `post_apply_attribution`, `AI_review`, `safety_or_broker_guard`, `user_authority`), owner artifact, observed evidence, next repair action, and the acceptance test that would move the blocker forward.

### 3.2 Automation-Chain Decision Principles

1. Intraday runtime threshold mutation is forbidden. Apply units are postclose report/calibration/AI review -> next preopen runtime env -> postclose attribution.
2. Unmet conditions close as calibration states, not rollback: `adjust_up`, `adjust_down`, `hold`, `hold_sample`, `hold_no_edge`, or `freeze`.
3. Rollback/safety revert is used only for hard/protect/emergency stop delay, order failure, provenance damage, same-stage owner conflict, or severe loss guard breach.
4. AI correction and pattern lab automation are reviewer/proposer/work-instruction layers. They have no standalone runtime apply authority without deterministic guard.
5. Runtime-loaded AI system/user prompts, reviewer prompts, JSON classifier prompts, and schema instructions use English ASCII as the default contract. Internal decision role descriptions, rules, JSON field descriptions, and fallback reasons remain English; Korean user-facing text is separated into a formatting contract. Korean briefing/Telegram text shown directly to users and contract-required canonical labels are exceptions. Future edits to prompt-contract areas must keep this English standard and must not reintroduce Korean internal prompt wording.
6. Code-improvement workorders may be consumed by unattended implementation automation when the work stays in source-quality, parser/schema, report, test, documentation, instrumentation, or sim/source-only improvement scope. A `runtime_effect=false` workorder is allowed to produce repo changes and validation artifacts without separate user approval, provided it does not change real runtime authority. User approval and real-runtime guards are required only when the implementation would convert sim/probe/source-only evidence into real order authority, PREOPEN live env selection, provider route, bot process state, cap release, broker/order guard behavior, or hard/protect/emergency safety behavior.
7. Existing fixed thresholds are not deleted; their roles change. Broker submit/stale quote/price freshness/stop/account/order/quantity/cooldown guards are `hard_safety`; `BUY_SCORE_THRESHOLD` and entry score cutoff/VPW/strength/momentum families are candidate-generation `baseline_prior`; score65_74/soft stop/holding/scale-in price guard are `bounded_tunable`; latency DANGER/stale/broker submit block is `hard_safety_submit_quality`; fallback/legacy latency/shadow axes are `legacy_archive`.
8. The runtime decision contract removes the assumption that higher score monotonically increases EV. Matrix may consume score as a feature, but score bucket itself must not be a fixed policy or standalone action authority.
9. LDM/ADM and `lifecycle_bucket_discovery` must not silently pass when bucket, runtime candidate, code-improvement workorder, or sim-auto approval is created but downstream consumers miss the handoff. If any surfaced bucket handoff is missing from `threshold_cycle_ev`, `runtime_approval_summary`, `code_improvement_workorder`, `runtime_apply_bridge`, or `threshold_cycle_postclose_verification`, postclose verification closes as `automation_handoff_gap`. When collection-layer source, bucket field/type, or dimension key is added/deleted/changed, discovery compares the LDM source contract snapshot and surfaces `source_contract_changes`; additions route to `new_bucket_candidate`, while deletions/field loss route to `code_patch_required` or source-quality blocker.
10. Canonical runtime label contracts are part of source quality. Prompt/schema producers may emit only declared English labels, and consumers must normalize only documented legacy aliases. Any undefined label is an explicit fail condition that blocks downstream promotion until the contract or producer is deliberately updated with tests.
11. Bucket tuning follows a daily-plus-cumulative separation: daily buckets are for new-pattern/source-quality/conflict-child detection, while parent-level sim/live authority must use rolling/MTD or post-apply version confirmation. Child combo buckets remain available for diagnosis, dominant pattern explanation, and exclusion-dimension candidates, but daily-only child EV or a single thin child sample must not become live authority. Parent granularity should avoid both one super-bucket and thin over-fragmentation; target-pass parent buckets are the promotion unit, and conflict children narrow the policy instead of blocking the entire parent by default.

### 3.3 Canary, Cohort, And Live Change Principles

1. Only one live canary is allowed within the same stage. Bounded multi-arm canaries within the same family are allowed only when arm-level provenance, allocation, and rollback guards are closed.
2. Canary-to-live conversion must close `N_min`, primary EV improvement, no rollback guard breach, applied/not-applied cohort comparison, no cross-contamination, and restart/rollback path.
3. New/supplemental alpha axes must not open as shadow-only. Support axes that do not change real order/exit decisions, such as transport/schema, report-only decision support, and counterfactual enrichment, are allowed only when documented as observe/report-only.
4. Sim/probe/combined/counterfactual authority must not be standalone evidence for real execution quality or broker order enablement. Swing has no phase0 real-canary approval path; actual trading reflection opens only through final full-live conversion approval after complete LDM parent bucket evidence, parsed review, source-quality gates, and explicit user approval artifact are closed.
